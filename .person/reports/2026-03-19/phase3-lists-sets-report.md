# Phase 3 Report: Lists & Sets (컬렉션 활용)

**Reporter**: G (Git User)
**Date**: 2026-03-19

## 1. 작업 요약
Redis의 컬렉션 데이터 타입인 **Lists**와 **Sets**를 활용하여 선입선출(FIFO) 기반의 활동 로그와 중복을 허용하지 않는 사용자 참여형 기능을 구현했습니다. 

## 2. 주요 변경 사항
### Lists: 최근 본 상품 (Activity Log)
- **명령어**: `LPUSH`, `LREM`, `LTRIM`, `LRANGE`
- **로직**: 상품 상세 조회 시 기존 기록을 삭제(`LREM`)한 후 최상단에 추가(`LPUSH`)하고, 최대 5개까지만 유지(`LTRIM`)하는 고정 크기 큐를 구현했습니다.
- **비유**: 마치 책상 위에 서류를 쌓을 때, 같은 서류가 있으면 빼서 맨 위에 올리고 너무 많아지면 맨 아래 서류를 버리는 것과 같습니다.

### Sets: 상품 좋아요 & 방문자 집계
- **명령어**: `SADD`, `SREM`, `SISMEMBER`, `SCARD`
- **로직**:
    - **좋아요**: 사용자 ID를 Set에 저장하여 한 사용자가 여러 번 누르는 것을 방지하고 실시간 카운트를 집계합니다.
    - **UV 집계**: 날짜별 키를 사용하여 일일 유니크 방문자 수를 중복 없이 기록합니다.
- **비유**: 출석부에 도장을 찍는 것과 같습니다. 같은 사람이 여러 번 도장을 찍어도 출석한 인원은 1명으로 계산되는 원리입니다.

### Web UI 및 아키텍처 개선
- **실시간성**: AJAX를 도입하여 페이지 새로고침 없이 좋아요 상태와 최근 본 상품 목록이 갱신되도록 개선했습니다.
- **통합**: `MonitoringController`의 기능을 `ProductController`로 통합하여 중복 매핑 문제를 해결하고 구조를 단순화했습니다.
- **안정성**: `ProductDto`에 `@Builder`와 `updateLikeInfo` 패턴을 도입하여 불변 객체의 데이터 처리를 안정화했습니다.

## 3. 검증 결과
- **방문자 수**: 동일 세션 새로고침 시 숫자가 증가하지 않음을 확인.
- **최근 본 상품**: 클릭 시 목록이 실시간 갱신되며, 최대 5개 유지를 확인.
- **좋아요**: 토글 동작 및 실시간 카운트 증감 확인.
- **애플리케이션 구동**: `Ambiguous mapping` 오류 해결 후 정상 기동 확인.

## 4. 향후 계획
- **Phase 4**: Sorted Sets(ZSet)를 활용하여 '인기 상품 랭킹' 실시간 리더보드 구현 예정.
