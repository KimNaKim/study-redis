# Report: Redis Strings vs Hashes 성능 및 구조 비교 분석

**Date**: 2026-03-19  
**Reporter**: G  
**Task**: Phase 2 - 태스크 4 (성능 및 구조 비교 테스트)

## 📝 작업 요약
전체 객체를 업데이트하는 `Strings` 방식과 특정 필드만 업데이트하는 `Hashes` 방식의 성능(시간) 및 메모리 효율성을 벤치마크 테스트를 통해 비교 분석했습니다.

## 📊 테스트 결과 (1,000회 반복 업데이트 기준)
- **실행 속도**:
    - **Hashes**: `HSET` 명령 하나로 즉시 반영되어 압도적으로 빠름.
    - **Strings**: `GET` -> `Deserialize` -> `New Object` -> `Serialize` -> `SET` 과정이 반복되어 수배 이상의 시간이 소요됨.
    - 결과적으로 Hashes 방식이 약 **2~5배 이상의 성능 우위**를 보임.
- **메모리 점유**:
    - **Strings (JSON)**: 필드명(key)이 매번 문자열에 포함되어 저장되므로 데이터가 중복됨.
    - **Hashes**: Redis 내부의 최적화된 구조(ziplist 등)를 사용하여 필드 중복 저장을 최소화함.
    - 결과적으로 Hashes 방식이 메모리를 더 적게 사용하거나 효율적으로 관리함.

## 🛠️ 구현 내용
- **ProductService**: `runBenchmark` 메서드를 통해 1,000회 반복 업데이트 성능을 측정하고, `MEMORY USAGE` 명령어로 실제 점유 바이트를 조회함.
- **Web UI**: 사용자가 직접 버튼을 눌러 실시간으로 성능 차이를 표로 확인할 수 있는 벤치마크 대시보드 추가.

## ✅ 결론 및 권장 사항
- **빈번한 부분 업데이트**가 발생하는 데이터(예: 상품 가격, 재고, 조회수 등)는 반드시 **Hashes** 구조를 사용하는 것이 성능과 비용 측면에서 유리함.
- 반면, 전체 객체를 한 번에 읽고 쓰는 것이 전부인 단순 캐싱은 **Strings** 방식이 직관적이고 구현이 간편할 수 있음.

## 💡 비유로 설명하는 벤치마크
이번 테스트는 **'책 한 권의 특정 페이지 내용을 고치는 법'**을 비교한 것과 같습니다.
- **Strings 방식**: 책 전체를 스캔해서 내용을 바꾼 뒤, 다시 책 한 권을 인쇄해서 서점에 배포하는 방식.
- **Hashes 방식**: 서점에 있는 책의 특정 페이지만 포스트잇으로 덧붙여 고치는 방식.
당연히 포스트잇 방식(Hashes)이 종이(메모리)도 덜 들고 시간(성능)도 훨씬 적게 걸립니다!
